What is jz?
jz (“javascript zero”) compiles the numeric core of JavaScript — math, DSP, parsers, tight loops — ahead-of-time to lean, GC-free WebAssembly. Nothing ships at runtime: the output is plain WASM that runs in any engine or standalone. The same source you’d run in Node, ~2.4× faster.
Is it really just JavaScript?
Valid jz is valid JS — every jz program runs and tests in any JavaScript engine unchanged, then compiles to WASM. jz is a focused subset (“the good parts” for compute), not a new language or a TypeScript-like dialect. You don’t port your code into it; you write JavaScript and compile the hot parts.
Why no type annotations?
Because let x: i32 isn’t valid JavaScript — annotations would break the promise that jz runs and tests unchanged in any engine. So jz infers types from what you already write: let bits = (a, b) => a | b pins i32 (a bitwise op); let half = (n) => n * 0.5 pins f64 (a non-integer literal). You never annotate, never leave JS — and that’s the sharp edge over AssemblyScript, which needs TypeScript-style types.
Can jz compile itself?
Yes — fully. jz’s whole pipeline (parse → infer → emit → encode) is itself compiled to WASM: dist/jz.wasm is jz compiled by jz. The CI gate test:self round-trips real programs through the wasm-hosted compiler and runs their output. The artifact that builds your code was built by jz.
How is it different from AssemblyScript?
AssemblyScript is a TypeScript-like dialect — you rewrite into it with explicit type annotations and a managed-memory runtime. jz compiles the JavaScript you already wrote, needs no annotations (it reads types off your literals and operators), ships no runtime or GC, and on our 16-kernel suite produces WASM ~2.3× faster than AssemblyScript at roughly the same size.
How fast is it, really?
On a 16-kernel bench (Apple M4 Max), jz averages 2.4× faster than the same source in V8, 1.1× faster than Rust → wasm, and 1.3× faster than Zig → wasm on numeric kernels — native is the peer, not the ceiling. Auto-vectorized reductions go further: a dot-product peaks at 13.5× over V8. Every number is reproducible — run the bench yourself.
What is jz good for — and not?
Good for numeric and math compute, DSP/audio, parsing and transforms, and standalone WASM utilities. Not for UI/frontend, backends/APIs, or async- and I/O-heavy logic — keep regular JavaScript there. jz is a sharp tool for the compute core, not a whole-app framework.
Can I use existing npm packages?
Only the ones that fit the subset. There’s no runtime, so packages touching the DOM, async/Promise, the network, or Node APIs won’t compile — but pure numeric, algorithmic, parser, or transform source does. Relative imports (./dep.js) bundle at compile time; bare specifiers resolve with --resolve (CLI) or { modules } (API). jz is for compiling your compute core, not the npm ecosystem.
How is jz different from plain JavaScript?
Small, documented divergences, all in service of no-runtime/no-GC: numbers are f64 (integer-proven values become i32 and wrap at ±2³¹); strings are UTF-8 bytes ("中".length is 3); objects are fixed-shape structs (a dot-key added after the literal is readable but doesn’t enumerate — use Map for dynamic keys); there’s no GC (call memory.reset() between batches). Run the same source under Node to catch any that touch your code.
Does it use SIMD?
Yes. jz auto-vectorizes fixed-count loops and reductions to WASM v128 — something a JIT won’t do from plain JS. Pass noSimd for engines without the SIMD proposal, or drop to explicit f32x4/i32x4 intrinsics when you want full control.
What JavaScript features are supported?
The compute subset: numbers, typed arrays, arrow functions and closures, control flow, and math/bitwise ops. Broader JS (var/function/class/==) is lowered automatically; strict mode enforces the pure canonical subset. Async, the DOM, and dynamic eval are out of scope by design.
How do I get started?
npm install jz, then jz`export let f = x => x*x` hands you an instantiated WASM module. Or edit the N-body sim above — it’s a jz program, recompiling on every keystroke. jz is experimental and pre-1.0, so pin a version.
